# v0.1.1 Release Notes

## Summary
v0.1.1 sharpens Hermes Rules Kit from a Hermes-only-feeling rules repo into a clearer execution framework with an explicit Hermes-native reference implementation.

This release strengthens three things:
- portability framing
- verification-contract enforcement
- failure-mode examples drawn from real agent mistakes

## Highlights

### 1. Clearer positioning
The repo now distinguishes between:
- **execution framework core** — agent-agnostic operating law
- **Hermes-native adapter** — concrete tool mappings, templates, and skills for Hermes

This makes the repo easier to adopt outside Hermes without pretending all runtimes are identical.

### 2. Stronger verification contract
The verification model is now more than a suggestion.

Changes include:
- expanded `docs/verification-contract.md`
- explicit minimum closeout structure
- updated `templates/closeout-template.md`
- README guidance that pushes contributors toward status-labeled delivery

Preferred labels remain:
- `changed`
- `verified`
- `unverified`
- `blocked`
- `assumption`
- `multimodal-grounded`

### 3. New portability documentation
A new `docs/portability.md` explains:
- what belongs in the transferable core
- what is Hermes-specific adapter behavior
- how to evaluate whether a rule is portable or runtime-bound

### 4. Failure-mode library
The repo now includes a dedicated `examples/failure-modes/` section covering mistakes like:
- unverified "fixed" claims
- stale plans after contradictory tool results
- symptom patches instead of invariant-owner fixes
- claiming a service is up without a health check
- context contamination across projects

These examples are meant to justify the rules with concrete before/after cases.

## Why this release matters
Before this release, the repo had strong ideas but left two adoption questions too implicit:
- which parts are universal versus Hermes-bound?
- is the status model a preference or a contract?

v0.1.1 answers both more clearly.

## Upgrade guidance
If you already use the kit:
1. read `docs/portability.md`
2. re-check your closeout format against `templates/closeout-template.md`
3. use `examples/failure-modes/` to teach contributors what the rules are protecting against

## Known limitations
- The repo still has only one fully documented runtime adapter: Hermes
- The verification contract is demonstrated through docs/templates/examples, not automated linting
- More failure modes and richer verifier examples are still needed for broader public maturity
